# Интеграция онлайн-записи с Яндекс Картами

## Контекст

Нужно подключить онлайн-запись в клинику через Яндекс Карты.

Важно: Яндекс рисует свой интерфейс записи внутри карточки организации. Мы не встраиваем наш виджет в Яндекс. Наша задача - дать Яндексу backend API, через который он сможет получить филиалы, услуги, врачей, свободные слоты и создать/перенести/отменить запись.

Текущая запись уже работает по цепочке:

```text
BookingWidgetV3 / текущий frontend -> adminzrenie-laravel -> 1С
```

Для Яндекса новые endpoint'ы добавляем в `adminzrenie-laravel`, а не в основной сайт `zrenie.clinic-7`.

## Бизнес-этап до разработки

Прямое API-подключение должен подтвердить Яндекс. Это не только техническая задача: Яндекс может не одобрить прямое подключение для одной клиники или небольшой сети и предложить подключение через партнера онлайн-записи.

Что нужно сделать руководству/менеджеру:

1. Связаться с Яндекс Бизнесом или менеджером Яндекса.
2. Подать заявку на подключение онлайн-записи через API.
3. Уточнить, можем ли мы подключиться напрямую.
4. Согласовать список филиалов, которые появятся с кнопкой записи в Яндекс Картах.
5. Получить от Яндекса `partnerName`, JWT secret / ключ подписи и требования к тестовому/production контуру.
6. Передать разработчику параметры интеграции и данные по филиалам.

Пока Яндекс не подтвердил прямое подключение, backend можно подготовить, но нельзя гарантировать запуск записи в production в Яндекс Картах.

## Как это мапится на наши сущности

Главное правило: карточка организации в Яндекс Картах соответствует конкретному адресу.

```text
Яндекс company      -> наш Branch
Яндекс resource     -> наш Doctor
Яндекс service      -> новая модель Service
Яндекс slot         -> OnecSlot или локальный слот DoctorShift
Яндекс booking      -> Application
```

То есть `companyId` в API Яндекса должен соответствовать `branches.id`, а не `clinics.id`.

Если у клиники несколько адресов, каждый адрес/филиал нужно сопоставить с отдельной карточкой организации в Яндекс Картах. Если врач принимает в нескольких филиалах, он должен отдаваться как resource в каждом подходящем филиале с расписанием именно этого филиала.

## Общая архитектура

```text
Пользователь
  -> Яндекс Карты / карточка филиала
  -> Yandex Booking API в adminzrenie-laravel
  -> существующая логика Application / слотов / 1С
  -> 1С
```

Новые endpoint'ы для Яндекса должны быть adapter-слоем поверх существующей бизнес-логики. Не нужно напрямую переиспользовать текущие HTTP-контроллеры как внутренние сервисы; лучше вынести общую логику в сервисы или аккуратно использовать уже существующие сервисы.

Что уже есть в проекте:

- `ApplicationController` - создание, перенос и отмена заявок.
- `OneCBookingService` - бронирование и отмена в 1С.
- `DoctorController@slots` - расчет слотов врача.
- `DoctorController@calendarAvailability` и `DoctorsByDateCalendarService` - календарь доступности.
- Модели `Branch`, `Clinic`, `Doctor`, `Application`, `OnecSlot`, `DoctorShift`.

## Что нужно реализовать в adminzrenie-laravel

### 1. Данные филиалов для Яндекса

Добавить в `branches` поля:

- `lat` - широта.
- `lon` - долгота.
- `rubrics` - JSON-массив рубрик Яндекса.
- `permalink` - идентификатор карточки организации в Яндекс Бизнесе / Яндекс Картах, если Яндекс его даст или менеджер сможет собрать.

`permalink` очень желателен: он упрощает матчинг филиала с карточкой в Яндекс Картах. Без него матчинг идет по названию, адресу, координатам и телефону.

### 2. Услуги

Добавить модель `Service` и таблицу `services`.

Минимальные поля:

- `id`
- `branch_id`
- `title`
- `description`
- `category`
- `price_from`
- `price_to`
- `duration_seconds`
- `status`

Нужна связь многие-ко-многим с врачами через `doctor_service`.

Услуга - обязательная часть сценария Яндекса. Текущий виджет может записывать сразу к врачу/в клинику, но Яндекс ожидает список услуг, по которым он сможет показывать запись пользователю.

### 3. Yandex Booking API

Префикс в нашем приложении:

```text
/api/v1/yandex
```

Планируемые endpoint'ы:

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/companies/feed` | Фид филиалов/организаций |
| GET | `/companies/{companyId}/services` | Услуги филиала |
| GET | `/companies/{companyId}/resources` | Врачи филиала |
| GET | `/companies/{companyId}/available_dates` | Даты, где есть свободные слоты |
| GET | `/companies/{companyId}/available_time_slots` | Свободные временные слоты |
| POST | `/bookings` | Создать запись |
| GET | `/bookings/{bookingId}` | Получить запись, если требуется спецификацией/проверкой Яндекса |
| PUT | `/bookings/{bookingId}` | Перенести/обновить запись |
| DELETE | `/bookings/{bookingId}` | Отменить запись |

Перед реализацией обязательно еще раз сверить точные пути, обязательные поля request/response и статусы ошибок с текущей спецификацией Яндекса.

### 4. Авторизация

Добавить middleware `YandexJwtAuth`.

Ожидаемая схема:

- Яндекс отправляет запросы с `Authorization: Bearer <JWT>`.
- Наш backend проверяет подпись JWT через `YANDEX_JWT_SECRET`.
- В конфиге хранится `YANDEX_PARTNER_NAME`.
- Payload JWT нужно проверять по требованиям конкретного endpoint'а.

Важно: руководству нужно получить не постоянный JWT-токен, а `partnerName` и JWT secret / ключ подписи. Токены будут приходить в запросах Яндекса.

В `composer.json` сейчас нет JWT-библиотеки. Для реализации, скорее всего, понадобится добавить пакет для проверки JWT или реализовать проверку через надежную существующую библиотеку.

### 5. Создание, перенос и отмена записей

`bookingId` лучше мапить на `applications.id`, но нужно сохранить внешний контекст Яндекса:

- источник записи: `yandex`
- `yandex_service_id`
- исходный payload Яндекса
- внешний/партнерский идентификатор брони, если Яндекс его передает

Хранить это можно в `integration_payload` или в отдельных полях, если понадобится удобный поиск/отладка.

Создание записи должно использовать существующую логику:

- создать `Application`
- указать `branch_id`, `clinic_id`, `doctor_id`, `cabinet_id`, `appointment_datetime`
- при 1С-режиме передать `onec_slot_id`
- вызвать текущую логику бронирования в 1С

Перенос:

- найти `Application` по `bookingId`
- проверить новый слот
- отменить/перебронировать в 1С через текущую логику
- вернуть ответ в формате Яндекса

Отмена:

- найти `Application`
- отменить в 1С, если запись была создана в 1С
- освободить локальный слот/обновить статус
- вернуть ответ в формате Яндекса

## Формат фида и справочников

Фид должен отдавать филиалы (`Branch`) как компании Яндекса.

В фиде нужны:

- `id` - наш `branches.id`
- `name` - название филиала/клиники
- `address`
- `phones`
- `lat`, `lon`, если есть
- `rubrics`
- `permalink`, если есть
- `services`

В услуге нужны:

- `id`
- `title`
- `durationSeconds`, если известна длительность
- `price` с `currencyCode: RUB`, если известна цена
- `resources`, если услуга доступна не у всех врачей

Цена может быть необязательной по API, но лучше передавать ее, если бизнес готов поддерживать актуальность.

Пагинация в API Яндекса должна быть cursor-based:

- `pagination.cursor`
- `pagination.hasMore`

Не использовать для Яндекса обычные `page/per_page`, если спецификация требует cursor.

## Что потребуется от руководства/Яндекса

До полноценной реализации нужны:

- подтверждение Яндекса, что прямое API-подключение разрешено;
- `YANDEX_PARTNER_NAME`;
- `YANDEX_JWT_SECRET`;
- требования к тестовому и production API;
- список филиалов, которые подключаем;
- ссылки/permalink карточек филиалов в Яндекс Картах;
- актуальные адреса, телефоны и координаты филиалов;
- рубрики Яндекса для филиалов;
- список услуг, цены и длительности;
- какие врачи оказывают какие услуги;
- правила записи: можно ли записывать без выбора конкретного врача, можно ли переносить/отменять, какие ограничения по времени.

## Важные риски

- Яндекс может не одобрить прямое API-подключение.
- Без `permalink` матчинг филиалов может занять дольше или потребовать ручного уточнения.
- Услуги и цены придется поддерживать актуальными в админке.
- Если врач принимает в нескольких филиалах, доступность нужно считать строго в рамках выбранного `companyId` / `Branch`.
- Формат ошибок и обязательные поля Яндекса нужно сверять с актуальной спецификацией непосредственно перед реализацией.
